# 良いコードとは何か？

良いコードには3つの属性があります。
- 動作すること
- パフォーマンスが良いこと
- 変更しやすいこと

## 動作すること

コードは意図したとおりに動作します。素晴らしい！

毎回動作しますか？同じ入力を与えたとき、毎回期待される出力を生成しますか？

システムに誤った入力を与えた場合はどうですか？それを処理しますか？それとも壊れますか？

特定の条件下で壊れることを意図している場合、期待どおりに壊れますか？

そして、もし壊れた場合、その理由がわかりますか？エラーがどこから来ているのかわかりますか？

## パフォーマンスが良いこと

あなたのシステムは十分に高速ですか？キビキビと動作しますか？

スケールアップした場合はどうでしょう？より多くの操作、より長いプロセスでも、依然として高速ですか？

もちろんハードウェアもタスクに対応している必要がありますが、あなたのコードはリソースを効率的に使用していますか？

例えば、コード内に`On^2`のような操作はありますか？`On^2`は、ループの中にさらにループがある場合のビッグオー表記です。

ネットワーク呼び出しを行う場合、それらをすべて一度に行っていますか？それとも必要なときにのみ行っていますか？

高価な計算についても同様に、どのように管理していますか？それらはソフトウェアの読み込みを遅らせますか？それともバックグラウンドで処理しますか？

## 変更しやすいこと

変更のしやすさ、別名保守性には2つのことが必要です。
- 読みやすいこと
- 十分にテストされていること

読みやすいとは、あなたのコードを読む人が誰でも、それがどのように機能するかについてかなり良いアイデアを持つことを意味します。
そのため、意図する動作を説明するためのいくつかのコメントを付け、できるだけシンプルに書かれるべきです。

また、ファイルをうまく整理することも役立ちます。

十分にテストされたコードとは、コードが意図したとおりに動作することを確認する方法があることを意味します。

私たちはしばしば、自動テストを使用してこれを行います。自動テストは、コードが意図したとおりに動作することを自動的にチェックする関数です。
そうすることで、現在のコードが意図したとおりに動作していることについて、高い信頼性を持つことができます。
また、誤った入力やエッジケースが発生した場合でも、それらを処理することを保証します。
そして、テストが整備されていることで、プロセス中に何も壊すことなく機能を更新できることが保証されます。
複雑なシステムでは、すべての部分がどのように連携しているかを常に把握しているわけではありません。

不十分にテストされたコードベースでは、「デグレード」（以前は動作していたものが壊れること）が発生することがよくあります。
