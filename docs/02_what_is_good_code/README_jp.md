# 良いコードとは何か？

良いコードには3つの特徴があります。
- 動作する
- パフォーマンスが高い
- 変更が容易である

## 動作する

そのコードはあなたが望むことを行います。素晴らしい！

それは毎回動作しますか？同じ入力を与えたとき、毎回期待される出力を生成しますか？

システムに誤った入力を与えた場合はどうでしょうか？それを処理しますか？それとも壊れますか？

特定の条件下で壊れることを望む場合、それは期待される方法で壊れますか？

そして、もしそれが壊れた場合、なぜ壊れたのか分かりますか？エラーがどこから来ているのか分かりますか？

## パフォーマンスが高い

あなたのシステムは十分に速いですか？キビキビしていますか？

スケールアップしたらどうなりますか？より多くの操作、より長いプロセスになった場合でも、まだ速いでしょうか？

ハードウェアがタスクに対応しているのは当然ですが、あなたのコードはそのリソースを効率的に使用していますか？

例えば、コード内に`On2`操作はありますか？`On2`は、ループの中のループに対するビッグオー記法です。

## 変更が容易である

変更の容易さ、つまり保守性には2つのことが必要です。
- 読みやすいコード
- 十分にテストされたコード

読みやすいとは、あなたのコードを読んだ誰もが、それがどのように機能するかをかなりよく理解できるということです。
したがって、それが何を意図しているかを説明するためのいくつかのコメントを添えて、できるだけ簡潔に書かれるべきです。
また、ファイルを適切に整理することも役立ちます。

十分にテストされたコードとは、コードが本来の動作をしていることを確認する手段があるということです。
私たちはこれを自動テストで行うことがよくあります。自動テストとは、私たちのコードが本来の動作を自動的に行っているかをチェックする関数です。
そのようにして、私たちのコードが今まさに望むことを行っているという高い信頼度を持つことができます。
誤った入力やエッジケースが発生した場合でも、それらを処理すること。
そして、テストがあることで、その過程で何も壊すことなく機能を更新できることが保証されます。
複雑なシステムでは、すべての部分がどのように連携しているかを常に知っているわけではありません。

テストが不十分なコードベースでは、「回帰（リグレッション）」、つまり以前は動作していたものが今は壊れているという事態がよく発生します。
